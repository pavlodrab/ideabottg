# Workflow: спеки и PR-маркеры

Это правило для всех изменений, которые ведутся по Kiro-спекам в `.kiro/specs/<feature>/`.

## tasks.md — единственный источник правды по прогрессу

В каждом PR, который реализует часть спеки, **обязательно** обновляется `tasks.md` в том же PR:

- Маркеры статуса задач:
  - `[ ]` — не начато
  - `[~]` — реализовано в открытом (не смерджённом) PR
  - `[x]` — смерджено в основную ветку
  - `[!]` — заблокировано (рядом — короткое объяснение)
- Рядом со статусом, отличным от `[ ]`, ставим ссылку на PR в формате `_(PR #N)_` или `_(PR #N — merged)_`.
- Если в ходе работы появились новые подзадачи — добавляем их в нужную фазу с `[ ]`. Не выкидываем и не переписываем историю.

## Шапка tasks.md

В начале файла поддерживаем две таблички:

1. **Прогресс по фазам** — для каждой фазы число `done / in PR / total` и общий статус.
2. **Открытые PR** — список незакрытых PR по этой спеке (номер, ветка, фаза, краткое описание).

Эти таблицы синхронизируются в том же PR, где меняются задачи.

## Не плодим housekeeping PR

Отдельный PR ради «обновить статусы в `tasks.md`» — анти-паттерн. Все изменения статусов идут вместе с реализацией. Единственное исключение — переключение `[~]` → `[x]` после мержа: можно сделать в следующем PR (вместе с его реализацией).

## Требования и дизайн

`requirements.md` и `design.md` правим только когда меняется содержание фичи — не на каждый PR. Если PR ломает существующие требования или меняет дизайн, обновляем эти файлы в том же PR.
